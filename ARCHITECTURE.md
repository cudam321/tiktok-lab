## Problem Statement

Managing 5 TikTok accounts fully ad-hoc with no tools, no tracking, and no system. Posting is manual (logging into each account one by one), analytics don't exist (no cross-account comparison, no historical data), and content decisions are guesswork. The entire workflow — posting, monitoring, learning from results — is broken because there's no infrastructure to support it.

## Demand Evidence

The builder is the user. They are personally managing these accounts daily and experiencing the pain firsthand. The pain is diffuse — posting, analytics, and content decisions are all manual and disconnected. No formal demand validation beyond self-use, but the builder has autonomy to build internal tools without approval gates.

Current state: fully ad-hoc. No tools, no spreadsheets, no tracking of any kind. Post and hope.

## Status Quo

Nothing. No tools, no spreadsheets, no system. TikTok native app 5 times over. This means:
- Zero baseline data on what content works
- No historical metrics to analyze
- No way to compare performance across accounts
- Every content decision is intuition, not evidence

The upside: any system is a massive improvement. The downside: the builder hasn't yet validated that systematic posting + analytics actually improves outcomes (assumed, not proven).

## Target User & Narrowest Wedge

**Target user:** The builder themselves — a self-directed operator managing 5 TikTok accounts within a company. Has autonomy to build internal tools, judged on results not process.

**Narrowest wedge:** Data accumulation + experiment tracking. This is what separates "just use Later" from "I have a system that learns." The wedge is not posting (Later does that) or basic analytics (Hootsuite does that) — it's the ability to run formal A/B tests across accounts and have an AI agent surface what's working and why.

## Constraints

- **5 accounts max** with add/remove capability
- **Zernio API** Build plan (120 req/min) with Analytics add-on
- **Locally hosted** — no cloud deployment, no webhooks (polling only)
- **AI agent is recommend-only** — never posts autonomously, always requires user confirmation
- **OpenMontage** for AI-powered video production pipeline (invoked by the agent for content production, not for reasoning/analysis)
- **Post failure retry policy**: 3 retries with exponential backoff (30s, 2min, 10min), then mark as permanently failed and surface to agent for pattern learning
- **SQLite backup**: Daily automated `.backup` to `data/backups/` with 30-day retention
- **OAuth flow**: Zernio OAuth redirects to `http://localhost:8000/auth/callback` (Zernio supports localhost redirects for development). Fallback: temporary Cloudflare Tunnel during initial account connection only.
- **Offline recovery**: APScheduler with `misfire_grace_time=3600` (1 hour). On startup, the poller catches up on any missed polling windows. If a scheduled post was missed while the machine was off, it surfaces an alert in the agent insights feed rather than auto-posting stale content.
- **TikTok API limitations via Zernio:**
  - Organic analytics: views, likes, comments, shares only (no reach/impressions)
  - Post metrics cached 60 min, follower stats refresh daily
  - 13.1% post failure rate via API
  - Content moderation stricter via API than native app
  - No inbox (comments/DMs) access
  - No post editing after publish
  - No access to TikTok music library

## Premises

1. **Building a custom posting/scheduling dashboard is worth the effort**, even though Later/Buffer already handle multi-account TikTok posting for 5 accounts. The value is in having everything — posting, analytics, experiments, agent — in one integrated system.

2. **The AI agent's primary value is analysis and recommendations** (what to post, when, why) — not content generation. OpenMontage is a Phase 2 enhancement, not the core value proposition.

3. **Formal experiment tracking is the differentiator.** Statistical A/B testing across hooks, text, edits, hashtags, posting times — this is what makes this tool worth building instead of using off-the-shelf.

4. **Data accumulation comes first.** The agent needs historical data to be useful. The first weeks of operation are about building the data foundation, not expecting AI insights.

5. **Local-only hosting is right for v1**, even though it means polling instead of webhooks and no mobile access.

### Database Schema

**Core tables:**
- `accounts` — zernio_id, profile_id, display_name, niche, health_status, connected_at
- `posts` — account_id, zernio_post_id, status (draft/producing/ready/scheduled/published/failed), caption, media_path, tiktok_settings, experiment_id, production_id
- `metric_snapshots` — post_id, captured_at, views, likes, comments, shares, computed engagement_rate = (likes + comments + shares) / views * 100 (time-series)
- `follower_snapshots` — account_id, date, count, growth_abs, growth_pct (daily)
- `experiments` — name, hypothesis, variable, variants (JSON), metric_target, min_sample_size, status, result_summary, confidence
- `experiment_assignments` — post_id, experiment_id, variant_name
- `agent_insights` — type (briefing/alert/suggestion/experiment_result), title, body, priority, is_read, is_acted_on
- `agent_conversations` — role, content, tool_calls (rolling window, last 20 messages)
- `agent_context_summaries` — week, account_id, summary_text (weekly structured summaries for agent context)

### Experiment Tracking System

Supported variables: hook_style, posting_time, hashtag_strategy, caption_style, edit_pace, video_length, content_type, text_overlay (and extensible).

**Statistical approach:** Default `min_sample_size` of 10 posts per variant. With realistic posting cadence (2-3 posts/week/account), a two-variant experiment on a single account takes 7-10 weeks to reach conclusion. Cross-account experiments are faster (5 accounts x 2-3 posts = 10-15 data points per week). Use Mann-Whitney U test for final comparison, with p < 0.05 as the significance threshold. For faster directional signals, the agent also reports Bayesian posterior probabilities after just 5 posts per variant, flagged as "preliminary" rather than "conclusive."

**Within-account sequential testing** is preferred over cross-account parallel testing for variables like posting_time and hook_style, since account audiences differ. Run ABAB cycles (variant A for 2 weeks, variant B for 2 weeks, repeated) on the same account to mitigate temporal confounds (trending topics, seasonal effects, algorithm changes). The agent flags external events (holidays, trending topics) that may explain variance. Cross-account parallel testing is better for content_type and hashtag_strategy experiments where audience differences are the point.

**Statistical power note:** With n=10 per variant, Mann-Whitney U has limited power (~40% for medium effect sizes). The Bayesian posterior is the primary decision tool; the frequentist test serves as a sanity check. The agent recommends larger samples when variance is high.

Agent-driven flow:
1. Agent proposes experiment based on data gaps
2. User approves, experiment enters "running" state
3. Each post is tagged with variant assignment
4. After 5 posts per variant, agent reports preliminary Bayesian signal
5. After min_sample_size (10) posts per variant, agent runs Mann-Whitney U comparison
6. Agent concludes with p-value and confidence interval
7. Results feed back into future suggestions

### Polling Strategy (Build Plan, 120 req/min)

| Job | Frequency | Requests/cycle |
|-----|-----------|---------------|
| Post metrics (last 30 days) | Every 30 min | ~20 (batch 20/req, ~4 req/account) |
| Follower stats | Daily midnight | 5 |
| Account health | Every 6 hours | 5 |
| Creator info (pre-post) | On-demand | 1 per post |
| Agent daily briefing | Daily 8am | 0 (local DB) |
| Agent anomaly scan | Every 2 hours | 0 (local DB) |

Total steady-state: ~54 requests/hour (~0.9/min). Well within 120/min limit. Zernio client implements token-bucket rate limiter; if limit is approached, on-demand requests queue behind scheduled polling. On 429, backoff per Zernio `Retry-After` header. All scheduled times use machine-local timezone.

### AI Agent Design

**Two modes:**
1. **Analyst** (always running) — queries local DB for performance data, generates daily briefings, flags anomalies, tracks experiments, recommends what/when/why to post
2. **Producer** (user-triggered) — invokes OpenMontage to produce video content based on insights, outputs to local drafts queue, user reviews and approves before posting

**Phase 5 core tools (8 tools, ships with agent):**
- `query_post_metrics` — SQL against metrics tables with date range, account, and aggregation params
- `query_follower_trends` — Growth analysis per account with period comparison
- `compare_experiments` — A/B test statistical comparison (Mann-Whitney U + Bayesian posterior)
- `get_account_health` — Per-account status, sync state, failure rates
- `get_content_calendar` — Scheduled posts, gaps, posting frequency
- `suggest_next_post` — Analytics + trends to actionable recommendation
- `suggest_experiment` — Proposes new A/B test based on data gaps and untested variables
- `flag_underperformers` — Surfaces accounts/content types trending below baseline

**Phase 6 extension tools (4 tools, ships with OpenMontage, v1.1):**
- `create_content_brief` — Generate production brief from insights
- `trigger_production` — Kick off OpenMontage pipeline
- `check_production_status` — Poll checkpoint state
- `list_drafts` — Show produced content awaiting review

**Anomaly detection definitions (agent anomaly scan, every 2 hours):**
- **Engagement drop**: Post engagement rate drops >50% compared to 7-day rolling average for that account
- **Follower loss**: Follower count decreases by >2% in 24 hours
- **Viral spike**: Post views exceed 5x the account's median within 6 hours of posting
- **Failure rate spike**: >30% of posts for an account failed in the last 7 days (above platform average)
- **Sync gap**: No new metric snapshots for an account in >2 hours (poller may be stuck)

**Agent context management:** Rolling window of last 20 messages for chat interactions. For proactive runs (briefings, anomaly scans), the agent receives a structured summary of key findings from the last 7 days (generated by a summarization pass over recent agent_insights), plus the current experiment states, rather than raw conversation history. Summaries are stored in a `agent_context_summaries` table with weekly granularity. This keeps token usage bounded regardless of system age.

**Trust boundary:** Agent can create content and write to local drafts, but NEVER publishes without explicit user confirmation. Status flow enforces this: draft -> ready -> (user approves) -> scheduled -> published.

### OpenMontage Integration

OpenMontage lives as a sibling directory. A thin FastAPI bridge module:
1. Writes a production brief (YAML) to OpenMontage's `projects/` directory
2. Triggers the relevant pipeline (clip-factory or animated-explainer for TikTok)
3. Monitors checkpoint files for completion
4. Moves rendered output to drafts queue

The agent produces the brief and triggers the pipeline but doesn't directly execute OpenMontage tools — it delegates to the bridge.

## Open Questions

1. **Agent model and cost**: Claude API calls for the agent (chat + daily briefings + anomaly scans) will accumulate costs. What's the monthly budget ceiling? This affects how aggressive the proactive analysis can be. Estimate: ~$20-50/month for Sonnet-based agent with daily briefings and moderate chat usage.

2. **OpenMontage asset sourcing** (v1.1): What video assets will OpenMontage use? Stock footage? AI-generated? Your own footage library? This determines which OpenMontage providers need API keys and what the per-video production cost looks like. Defer until Phase 6.

3. **TikTok account association risk**: TikTok monitors account association. Running 5 accounts through the same Zernio API key from the same IP... is there risk of triggering TikTok's anti-association detection? Research indicates this is a real concern for multi-account operators. Mitigation: Zernio acts as an intermediary (calls originate from their servers, not yours), which should reduce association risk. Monitor for warnings.

4. **Content moderation feedback loop**: The agent should track `failure_reason` on failed posts and, after accumulating 10+ failures, surface pattern analysis (e.g., "3 of your last 5 failures on Account X mentioned 'community guidelines' — consider reviewing content themes for that account").

## The Assignment

Before you write a single line of code: **Go post 5 pieces of content this week — one per account — and manually track what you posted, when, and why.** Write it down in a spreadsheet or even a text file. This does two things: (1) it gives you the first data points your agent will eventually reason over, and (2) it forces you to articulate your content strategy in words, which becomes your agent's initial system prompt. You can't build a system to optimize something you haven't yet described.
